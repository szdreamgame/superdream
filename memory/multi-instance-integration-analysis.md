## 三、整合方案

### 方案 A: 将另一个 OpenClaw 配置为子智能体

**核心思路**: 将另一个实例作为"远程子智能体"注册到超梦的管理系统中。

#### 实施步骤

1. **在超梦配置中添加远程智能体**

   编辑 `/root/.openclaw/openclaw.json`:
   
   ```json
   {
     "agents": {
       "list": [
         // ... 现有智能体
         {
           "id": "remote-instance",
           "name": "远程实例",
           "model": "dashscope-coding/qwen3.5-plus",
           "remote": {
             "enabled": true,
             "url": "wss://<另一个实例的 IP>:<端口>",
             "token": "<认证 token>"
           }
         }
       ]
     }
   }
   ```

2. **配置消息路由**

   在飞书插件中配置特定群聊或命令路由到远程实例:
   
   ```json
   {
     "channels": {
       "feishu": {
         "agents": {
           "oc_xxx_remote_chat": "remote-instance"
         }
       }
     }
   }
   ```

3. **配置 Gateway 远程连接**

   另一个实例需要配置 Gateway 连接到超梦:
   
   ```json
   {
     "gateway": {
       "remote": {
         "url": "wss://47.102.199.115:15427",
         "token": "bde3190da2271b03d3316b25f43899fb"
       }
     }
   }
   ```

#### 优点

- ✅ 统一管理界面 (Pixel Office 可显示所有智能体)
- ✅ 共享配置和凭证 (飞书 App、模型 API 等)
- ✅ 集中式日志和监控
- ✅ 智能体间可直接通信
- ✅ 资源利用优化 (可动态分配任务)

#### 缺点

- ❌ 需要修改 OpenClaw 核心代码支持远程智能体
- ❌ 网络延迟可能影响响应速度
- ❌ 单点故障风险 (超梦实例宕机影响所有)
- ❌ 配置复杂度增加

#### 技术可行性

**中等** - 需要 OpenClaw 支持远程智能体注册和消息路由机制。当前架构中智能体是本地进程，需要扩展为支持远程 RPC 调用。

---

### 方案 B: 统一群聊路由到当前实例

**核心思路**: 将所有飞书群聊的消息统一路由到超梦实例处理，另一个实例仅作为备份或特殊用途。

#### 实施步骤

1. **飞书 Bot 配置统一**

   确保所有群聊使用的飞书 Bot 都指向超梦实例:
   
   - 检查所有群聊的 Bot 配置
   - 统一使用相同的 App ID (cli_a92e8b3399b85cd6)
   - 配置 webhook 指向超梦 Gateway

2. **配置消息路由规则**

   在 `/root/.openclaw/workspace/` 下创建路由配置:
   
   ```json
   {
     "routing": {
       "mode": "chat_id",
       "routes": [
         {
           "chat_id": "oc_95a9882e1aca9546c1930b2d27660a6a",
           "agent": "yiwen",
           "commands": ["/文字", "/转写"]
         },
         {
           "chat_id": "oc_f22ffe36d557729c0d77f8b11c74e0bd",
           "agent": "huayu",
           "commands": ["/生成", "/画"]
         },
         {
           "chat_id": "*",
           "agent": "main",
           "default": true
         }
       ]
     }
   }
   ```

3. **配置另一个实例为热备份**

   ```json
   {
     "gateway": {
       "ha": {
         "enabled": true,
         "peer": {
           "url": "wss://47.102.199.115:15428",
           "heartbeat_interval": 30000,
           "failover_timeout": 60000
         }
       }
     }
   }
   ```

#### 优点

- ✅ 架构简单，无需大改
- ✅ 消息处理集中，便于管理
- ✅ 支持故障转移 (HA)
- ✅ 飞书配置简单 (单一 Bot)

#### 缺点

- ❌ 另一个实例大部分时间闲置
- ❌ 无法利用另一个实例的计算资源
- ❌ 单点处理瓶颈
- ❌ 群聊数量受单个实例性能限制

#### 技术可行性

**高** - 当前架构已支持基于 chat_id 的路由，只需配置即可实现。HA 功能需要额外开发。

### 方案 C: 跨实例消息转发

**核心思路**: 两个实例独立运行，通过消息转发机制实现协同工作。

#### 实施步骤

1. **建立实例间通信通道**

   创建消息转发服务:
   
   ```javascript
   // /root/.openclaw/workspace/message-bridge.js
   const WebSocket = require('ws');
   
   class InstanceBridge {
     constructor(config) {
       this.peers = config.peers; // [{url, token, agents}]
       this.ws = null;
     }
     
     async connect() {
       // 连接到对等实例
     }
     
     async forwardMessage(message) {
       // 转发消息到目标实例
     }
   }
   ```

2. **配置消息转发规则**

   ```json
   {
     "messageBridge": {
       "enabled": true,
       "peers": [
         {
           "name": "实例 2",
           "url": "wss://<IP>:<PORT>",
           "token": "<token>",
           "forwardRules": [
             {
               "fromChat": "oc_xxx",
               "toAgent": "remote-agent",
               "conditions": ["has_video", "command:/特殊"]
             }
           ]
         }
       ]
     }
   }
   ```

3. **在飞书插件中集成转发钩子**

   修改 `bot.ts` 添加转发逻辑:
   
   ```typescript
   // 在 handleFeishuMessage 中
   if (shouldForwardToPeer(message)) {
     await messageBridge.forward(message);
     return; // 不继续本地处理
   }
   ```

4. **配置去重机制**

   确保转发的消息不会被重复处理:
   
   ```json
   {
     "feishu": {
       "dedup": {
         "crossInstance": true,
         "sharedRedis": "redis://localhost:6379"
       }
     }
   }
   ```

#### 优点

- ✅ 两个实例完全独立，互不影响
- ✅ 可灵活分配不同群聊/功能到不同实例
- ✅ 负载均衡 (可按群聊或功能拆分)
- ✅ 易于扩展 (可添加更多实例)

#### 缺点

- ❌ 架构复杂度高
- ❌ 需要额外的消息桥接服务
- ❌ 去重机制复杂 (需要共享存储)
- ❌ 调试困难 (问题可能出现在任一实例)
- ❌ 消息延迟增加 (转发开销)

#### 技术可行性

**中等** - 需要开发消息桥接服务和跨实例去重机制。WebSocket 通信成熟，但集成到现有架构需要工作量。

## 七、风险与注意事项

### 7.1 安全风险

- ⚠️ **Token 泄露**: Gateway token 需要妥善保管
- ⚠️ **飞书 App Secret**: 不要提交到代码仓库
- ⚠️ **网络暴露**: Gateway 端口不要直接暴露在公网

### 7.2 性能风险

- ⚠️ **单点瓶颈**: 所有消息集中到一个实例
- ⚠️ **网络延迟**: 跨实例通信增加延迟
- ⚠️ **资源竞争**: 多个智能体共享资源

### 7.3 运维风险

- ⚠️ **配置同步**: 确保两个实例配置一致
- ⚠️ **版本兼容**: OpenClaw 版本需要保持一致
- ⚠️ **升级协调**: 升级时需要协调两个实例
